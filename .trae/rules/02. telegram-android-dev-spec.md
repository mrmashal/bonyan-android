# Telegram Android — Internal Development Specification

> Source of truth for AI-assisted development in this repository.
> All new code must conform to the patterns documented here.

---

## 1. Repository Overview

### High-Level Architecture

This is the official Telegram Android client. It is a single-process Android application with a custom navigation stack, a custom MTProto network layer, a custom SQLite wrapper, and a fully custom UI toolkit built on top of raw Android Views (no Jetpack Compose, no XML layouts for screens).

### Module Structure

| Module | Role |
|---|---|
| `TMessagesProj` | Core library: all business logic, UI, network, storage |
| `TMessagesProj_App` | Google Play release flavor |
| `TMessagesProj_AppHockeyApp` | Beta/HockeyApp flavor |
| `TMessagesProj_AppHuawei` | Huawei AppGallery flavor |
| `TMessagesProj_AppStandalone` | Standalone/direct APK flavor |

All feature code lives in `TMessagesProj/src/main/java/org/telegram/`.

### Key Package Map

| Package | Contents |
|---|---|
| `org.telegram.messenger` | Controllers, singletons, utilities, background services |
| `org.telegram.tgnet` | MTProto network layer, TL schema, ConnectionsManager |
| `org.telegram.SQLite` | Custom SQLite wrapper (not Room, not standard Android SQLite API) |
| `org.telegram.ui` | All screens (fragments) |
| `org.telegram.ui.ActionBar` | Navigation, ActionBar, BaseFragment, Theme, dialogs |
| `org.telegram.ui.Cells` | RecyclerView cell classes |
| `org.telegram.ui.Components` | Reusable UI components, alerts, bottom sheets |
| `org.telegram.ui.Adapters` | RecyclerView adapters |

## 2. UI/UX Development Rules

### 2.1 Screen Creation Rules

**Must follow: every new screen extends `BaseFragment`.**

- File: `org.telegram.ui.ActionBar.BaseFragment`
- Never use Android `Fragment`, `Activity` (except `LaunchActivity`), or Jetpack Navigation.
- Never use XML layout files for screens. All layouts are built programmatically in `createView(Context context)`.

**Canonical screen skeleton:**

```java
public class MyFeatureActivity extends BaseFragment implements NotificationCenter.NotificationCenterDelegate {

    // Row index fields (one per list row type)
    private int myFirstRow;
    private int mySecondRow;
    private int rowCount;

    private RecyclerListView listView;
    private ListAdapter adapter;

    @Override
    public boolean onFragmentCreate() {
        super.onFragmentCreate();
        // Subscribe to NotificationCenter events
        getNotificationCenter().addObserver(this, NotificationCenter.someEvent);
        updateRows(); // compute row indices
        return true;
    }

    @Override
    public void onFragmentDestroy() {
        super.onFragmentDestroy();
        // Unsubscribe — always mirror addObserver calls
        getNotificationCenter().removeObserver(this, NotificationCenter.someEvent);
    }

    @Override
    public View createView(Context context) {
        actionBar.setBackButtonDrawable(new BackDrawable(false));
        actionBar.setTitle(getString(R.string.MyFeatureTitle));
        actionBar.setActionBarMenuOnItemClick(new ActionBar.ActionBarMenuOnItemClick() {
            @Override
            public void onItemClick(int id) {
                if (id == -1) finishFragment();
            }
        });

        FrameLayout contentView = new FrameLayout(context);
        contentView.setBackgroundColor(Theme.getColor(Theme.key_windowBackgroundGray));

        listView = new RecyclerListView(context);
        listView.setLayoutManager(new LinearLayoutManager(context));
        adapter = new ListAdapter(context);
        listView.setAdapter(adapter);
        contentView.addView(listView, LayoutHelper.createFrame(LayoutHelper.MATCH_PARENT, LayoutHelper.MATCH_PARENT));

        return fragmentView = contentView;
    }

    private void updateRows() {
        rowCount = 0;
        myFirstRow = rowCount++;
        mySecondRow = rowCount++;
    }

    @Override
    public void didReceivedNotification(int id, int account, Object... args) {
        if (id == NotificationCenter.someEvent) {
            updateRows();
            if (adapter != null) adapter.notifyDataSetChanged();
        }
    }
    // ... ListAdapter inner class (see Section 2.5)
}
```

**Lifecycle order:** `onFragmentCreate` → `createView` → `onResume` → `onPause` → `onFragmentDestroy`

- Register `NotificationCenter` observers in `onFragmentCreate`, remove them in `onFragmentDestroy`.
- Cancel network requests automatically: `onFragmentDestroy` in `BaseFragment` calls `getConnectionsManager().cancelRequestsForGuid(classGuid)`.
- Use `classGuid` (auto-assigned in `BaseFragment` constructor) to tag all network requests made by a fragment.

### 2.2 Navigation Rules

**Must follow: use `presentFragment` / `finishFragment` for all navigation.**

```java
// Push a new screen
presentFragment(new MyFeatureActivity());

// Push and remove current screen from stack
presentFragment(new MyFeatureActivity(), true);

// Go back
finishFragment();

// Remove self without animation
removeSelfFromStack();
```

- `presentFragment` is a method on `BaseFragment` — call it directly inside any fragment.
- Never start an `Activity` for navigation between app screens. `LaunchActivity` is the single host activity.
- Pass data between screens via `Bundle arguments`:

```java
Bundle args = new Bundle();
args.putLong("dialog_id", dialogId);
MyFeatureActivity fragment = new MyFeatureActivity(args);
// In the target fragment:
long dialogId = getArguments().getLong("dialog_id");
```

### 2.3 ActionBar Rules

**Must follow: configure `actionBar` inside `createView`, not in a separate method.**

```java
actionBar.setBackButtonDrawable(new BackDrawable(false));
actionBar.setTitle(getString(R.string.MyTitle));
actionBar.setAllowOverlayTitle(true); // for collapsible titles

// Adding menu items:
ActionBarMenu menu = actionBar.createMenu();
menu.addItem(BUTTON_ID, R.drawable.msg_search);

// Handling clicks:
actionBar.setActionBarMenuOnItemClick(new ActionBar.ActionBarMenuOnItemClick() {
    @Override
    public void onItemClick(int id) {
        if (id == -1) {          // -1 is always the back button
            finishFragment();
        } else if (id == BUTTON_ID) {
            // handle
        }
    }
});
```

- `actionBar` is provided by `BaseFragment` — never instantiate `ActionBar` manually in a fragment.
- Colors must use `Theme.getColor(Theme.key_*)` — never hardcode colors.
- Use `getThemedColor(Theme.key_*)` inside fragments (delegates to `resourceProvider` if set).

### 2.4 Dialog and Popup Rules

**Rule: never use `android.app.AlertDialog` directly. Always use `org.telegram.ui.ActionBar.AlertDialog`.**

**Simple alert:**
```java
AlertDialog.Builder builder = new AlertDialog.Builder(getParentActivity());
builder.setTitle(getString(R.string.AppName));
builder.setMessage(getString(R.string.SomeMessage));
builder.setPositiveButton(getString(R.string.OK), null);
builder.setNegativeButton(getString(R.string.Cancel), null);
showDialog(builder.create()); // use showDialog(), not dialog.show()
```

**Loading dialog:**
```java
AlertDialog progressDialog = new AlertDialog(getParentActivity(), AlertDialog.ALERT_TYPE_SPINNER);
showDialog(progressDialog);
// dismiss later:
progressDialog.dismiss();
```

**Rule: always call `showDialog(dialog)` from `BaseFragment`, not `dialog.show()` directly.** This ensures proper lifecycle tracking via `visibleDialog`.

**Bottom sheets — use `BottomSheet` or `BottomSheetWithRecyclerListView`:**

```java
// Simple list bottom sheet:
BottomSheet.Builder builder = new BottomSheet.Builder(getParentActivity());
builder.setItems(new CharSequence[]{item1, item2}, (dialog, which) -> { ... });
showDialog(builder.create());

// Rich bottom sheet with RecyclerView:
// Extend BottomSheetWithRecyclerListView
// Location: org.telegram.ui.Components.BottomSheetWithRecyclerListView
```

**Context menus / item options — use `ItemOptions`:**
```java
ItemOptions.makeOptions(fragment, anchorView)
    .add(R.drawable.msg_edit, getString(R.string.Edit), () -> { ... })
    .add(R.drawable.msg_delete, getString(R.string.Delete), true, () -> { ... }) // true = red/destructive
    .show();
```
- Location: `org.telegram.ui.Components.ItemOptions`
- Use this instead of `PopupWindow` or `ActionBarPopupWindow` for item-level context menus.

**Common dialogs — check `AlertsCreator` first:**
- Location: `org.telegram.ui.Components.AlertsCreator`
- Contains factory methods for: clear history, delete dialog, report, ban user, mute, etc.
- Before writing a new confirmation dialog, search `AlertsCreator` for an existing one.

### 2.5 List / RecyclerView Rules

**Must follow: use `RecyclerListView` instead of `RecyclerView`.**
- Location: `org.telegram.ui.Components.RecyclerListView`
- Provides: section headers, fast scroll, item click/long-click listeners, selection mode.

**Standard adapter pattern (classic, used in most settings screens):**

```java
private class ListAdapter extends RecyclerView.Adapter<RecyclerView.ViewHolder> {
    private Context mContext;

    public ListAdapter(Context context) { mContext = context; }

    @Override
    public int getItemCount() { return rowCount; }

    @Override
    public int getItemViewType(int position) {
        if (position == myTextRow) return 0;
        if (position == myCheckRow) return 1;
        return 2;
    }

    @Override
    public RecyclerView.ViewHolder onCreateViewHolder(ViewGroup parent, int viewType) {
        View view;
        if (viewType == 0) {
            view = new TextCell(mContext, resourceProvider);
        } else if (viewType == 1) {
            view = new TextCheckCell(mContext, resourceProvider);
        } else {
            view = new ShadowSectionCell(mContext);
        }
        view.setLayoutParams(new RecyclerView.LayoutParams(
            RecyclerView.LayoutParams.MATCH_PARENT, RecyclerView.LayoutParams.WRAP_CONTENT));
        return new RecyclerListView.Holder(view);
    }

    @Override
    public void onBindViewHolder(RecyclerView.ViewHolder holder, int position) {
        if (position == myTextRow) {
            ((TextCell) holder.itemView).setTextAndIcon(getString(R.string.Label), R.drawable.icon, false);
        } else if (position == myCheckRow) {
            ((TextCheckCell) holder.itemView).setTextAndCheck(getString(R.string.Label), someBoolean, true);
        }
    }
}
```

**Modern adapter pattern — use `UniversalAdapter` + `UItem` for new settings-style screens:**
- Location: `org.telegram.ui.Components.UniversalAdapter`, `UItem`
- Extend `UniversalFragment` for the simplest path:

```java
public class MyFragment extends UniversalFragment {
    @Override protected CharSequence getTitle() { return getString(R.string.MyTitle); }

    @Override
    protected void fillItems(ArrayList<UItem> items, UniversalAdapter adapter) {
        items.add(UItem.asHeader(getString(R.string.SectionHeader)));
        items.add(UItem.asCheck(0, getString(R.string.ToggleLabel)).setChecked(someBoolean));
        items.add(UItem.asText(1, getString(R.string.NavigateLabel)));
        items.add(UItem.asShadow(null));
    }

    @Override
    protected void onClick(UItem item, View view, int position, float x, float y) {
        if (item.id == 0) { /* toggle */ }
        else if (item.id == 1) { presentFragment(new OtherFragment()); }
    }

    @Override
    protected boolean onLongClick(UItem item, View view, int position, float x, float y) {
        return false;
    }
}
```

**Set click listeners on `RecyclerListView`, not on individual cells:**
```java
listView.setOnItemClickListener((view, position) -> { ... });
listView.setOnItemLongClickListener((view, position) -> { ...; return true; });
```

### 2.7 Theming and Styling Rules

```java
// Inside a fragment (preferred — respects resourcesProvider):
int color = getThemedColor(Theme.key_windowBackgroundWhite);

// Inside a cell or component (pass resourcesProvider through constructor):
int color = Theme.getColor(Theme.key_windowBackgroundWhiteBlackText, resourcesProvider);

// Inside a static context:
int color = Theme.getColor(Theme.key_actionBarDefault);
```

| Key | Usage |
|---|---|

| `Theme.key_windowBackgroundGray` | Gray screen background |
| `Theme.key_windowBackgroundWhiteBlackText` | Primary text on white |

| `Theme.key_actionBarDefault` | ActionBar background |

| `Theme.key_switchTrack` / `key_switchTrackChecked` | Toggle switch colors |

**Dimensions — always use `AndroidUtilities.dp(float)` for density-independent pixels:**
```java
import static org.telegram.messenger.AndroidUtilities.dp;
// Usage:
view.setPadding(dp(16), dp(8), dp(16), dp(8));
```

**Layout params — always use `LayoutHelper`:**
```java
import org.telegram.ui.Components.LayoutHelper;
// FrameLayout child:
addView(child, LayoutHelper.createFrame(LayoutHelper.MATCH_PARENT, 50, Gravity.TOP, 0, 0, 0, 0));
// LinearLayout child:
addView(child, LayoutHelper.createLinear(LayoutHelper.MATCH_PARENT, LayoutHelper.WRAP_CONTENT));
```
- `LayoutHelper.MATCH_PARENT` = -1, `LayoutHelper.WRAP_CONTENT` = -2
- Margin values passed to `LayoutHelper.createFrame` are in dp — `LayoutHelper` calls `dp()` internally.

**RTL support — always use `LocaleController.isRTL` for gravity:**
```java
textView.setGravity(LocaleController.isRTL ? Gravity.RIGHT : Gravity.LEFT);
```

### 2.8 Localization Rules

**Must follow: never hardcode user-visible strings. Always use `R.string` resources.**

```java
import static org.telegram.messenger.LocaleController.getString;
import static org.telegram.messenger.LocaleController.formatString;
import static org.telegram.messenger.LocaleController.formatPluralString;

// Simple string:
String s = getString(R.string.Cancel);

// Formatted string:
String s = formatString(R.string.SomeFormat, arg1, arg2);

// Plural:
String s = formatPluralString("Chats", count); // uses plural rules
```

- String resources are in `TMessagesProj/src/main/res/values/strings.xml`.
- Telegram uses its own plural system via `LocaleController` — use `formatPluralString`, not `getResources().getQuantityString`.

### 2.9 Notification / Toast Rules

**Must follow: use `BulletinFactory` for in-app toast-style notifications. Never use `android.widget.Toast`.**

```java
// Simple text bulletin:
BulletinFactory.of(this).createSimpleBulletin(R.raw.some_lottie, getString(R.string.Done)).show();

// Error bulletin:
BulletinFactory.of(this).showForError(error);

// From a BottomSheet:
BulletinFactory.of(bottomSheet.container, resourcesProvider).createSimpleBulletin(...).show();
```
- Location: `org.telegram.ui.Components.BulletinFactory`
- `Bulletin` auto-dismisses after a timeout and stacks properly with the navigation layout.

### 2.10 Layout Positioning Below the ActionBar

This is one of the most common sources of bugs in new screens. The ActionBar is **not** a child of `fragmentView` — it is a sibling drawn by the framework on top of it. Any view added to `fragmentView` with `Gravity.TOP` starts at `y=0`, which is directly behind the ActionBar and will be invisible.

#### Rule: always use `SizeNotifierFrameLayout` as the root view

`SizeNotifierFrameLayout` (in `org.telegram.ui.Components`) fires `onInsets()` on the fragment when system insets change (navigation bar, keyboard). Use it as `fragmentView` for any screen that needs correct bottom padding or keyboard handling.

```java
fragmentView = contentView = new SizeNotifierFrameLayout(context) {
    @Override
    protected void onMeasure(int widthMeasureSpec, int heightMeasureSpec) {
        measureChildWithMargins(actionBar, widthMeasureSpec, 0, heightMeasureSpec, 0);
        // Update layout-dependent values here, before super.onMeasure
        checkUi_listViewPadding();
        super.onMeasure(widthMeasureSpec, heightMeasureSpec);
    }

    @Override
    protected void onLayout(boolean changed, int left, int top, int right, int bottom) {
        super.onLayout(changed, left, top, right, bottom);
        // Apply dynamic translations here, after super.onLayout
        checkUi_stickyHeaderY();
        checkUi_floatingButtonPosition();
    }
};
```

**Critical:** call `measureChildWithMargins(actionBar, ...)` as the **first line** of `onMeasure` so that `actionBar.getMeasuredHeight()` returns the correct value for all subsequent calculations in the same pass.

#### Three positioning patterns — use the right one for each case

**Pattern 1 — `LayoutParams.topMargin` (for views with fixed offset below ActionBar)**

Use in `onMeasure`, before `super.onMeasure`. The layout system then places the view at the correct `y` with correct bounds. This is the only safe approach — `setTranslationY` moves the view visually but leaves its layout bounds at `y=0`, which causes the parent to clip it when `clipChildren=true`.

```java
// In onMeasure:
measureChildWithMargins(actionBar, widthMeasureSpec, 0, heightMeasureSpec, 0);
((MarginLayoutParams) stickyView.getLayoutParams()).topMargin = actionBar.getMeasuredHeight();
// If the sticky view itself has a dynamic height (e.g. animated expand/collapse),
// update the margin every frame from the animation's addUpdateListener.
super.onMeasure(widthMeasureSpec, heightMeasureSpec);
```

Real examples: `ContactsActivity` (emptyView, headerShadowView), `PlannerActivity` (CalendarHeaderView).

**Pattern 2 — `setTranslationY` (for views that animate or scroll)**

Use in `onLayout` or during scroll/animation callbacks. Only safe when the parent has `clipChildren=false`, or when the view stays within its layout bounds.

```java
// In onLayout (or a checkUi_ method called from onLayout):
stickyView.setTranslationY(actionBar.getMeasuredHeight());

// During scroll (DialogsActivity pattern — accumulate heights):
float totalOffset = actionBar.getMeasuredHeight() + scrollYOffset;
filterTabsView.setTranslationY(totalOffset);
totalOffset += filterTabsView.getMeasuredHeight() * filterTabsView.getAlpha();
topPanelLayout.setTranslationY(totalOffset);
```

Real examples: `ContactsActivity` (searchField), `DialogsActivity` (filterTabsView, topPanelLayout), `InviteContactsActivity` (searchField).

**Pattern 3 — `listView.setPadding` (for the scrollable list itself)**

Always use `setClipToPadding(false)` alongside this so content can scroll under the padding area (e.g. behind the ActionBar on overscroll).

```java
private void checkUi_listViewPadding() {
    listView.setPadding(
        0,
        actionBar.getMeasuredHeight() + dp(extraHeaderHeightDp),  // top
        0,
        navigationBarHeight + additionNavigationBarHeight          // bottom
    );
}
// Called from onMeasure (before super) and from onInsets().
```

Real examples: `ContactsActivity`, `InviteContactsActivity`, `NotificationsSettingsActivity` (via `setAdaptiveBackground`).

#### Handling system insets (`navigationBarHeight`)

Override `onInsets` on the fragment to receive bottom inset updates, then recalculate padding:

```java
private int navigationBarHeight;
private int additionNavigationBarHeight; // dp(MAIN_TABS_HEIGHT_WITH_MARGINS) when hasMainTabs

@Override
public void onInsets(int left, int top, int right, int bottom) {
    navigationBarHeight = bottom;
    checkUi_listViewPadding();
    checkUi_floatingButtonPosition();
}
```

When the screen is shown as a main tab (`hasMainTabs = true`), add extra bottom space for the tab bar:

```java
// In onFragmentCreate:
additionNavigationBarHeight = hasMainTabs
    ? dp(DialogsActivity.MAIN_TABS_HEIGHT_WITH_MARGINS) : 0;
additionFloatingButtonOffset = hasMainTabs
    ? dp(DialogsActivity.MAIN_TABS_HEIGHT + DialogsActivity.MAIN_TABS_MARGIN) : 0;

// FAB lift:
floatingButton.setTranslationY(-(navigationBarHeight + additionFloatingButtonOffset));
```

#### Anti-patterns — never do these

| Anti-pattern | Why it breaks | Correct alternative |
|---|---|---|
| `view.setTranslationY(abH)` with `clipChildren=true` parent | View is clipped at `y=0` layout bounds | Set `LayoutParams.topMargin` in `onMeasure` instead |
| Hardcode `dp(56)` for ActionBar height | Varies by device, density, and overlay mode | Always use `actionBar.getMeasuredHeight()` |
| Set `topMargin` in `onLayout` | Layout pass already ran; margin takes effect next frame | Set in `onMeasure` before `super.onMeasure` |
| `view.post(() -> setPadding(...))` | Runs after first draw; causes visible jump | Use `onMeasure` override instead |
| `AndroidUtilities.statusBarHeight` for list top padding | Misses ActionBar height on non-overlay ActionBar screens | Use `actionBar.getMeasuredHeight()` |

### 2.11 FragmentFloatingButton — Correct Usage

`FragmentFloatingButton` is **not** a plain `View`. It manages its own visibility state via an internal `BoolAnimator`. Simply adding it to the layout is not enough — it starts invisible and must be explicitly shown.

#### Required setup

```java
// 1. Create and add with the standard layout params
floatingButton = new FragmentFloatingButton(context, resourceProvider);
floatingButton.setImageResource(R.drawable.ic_ab_new);
floatingButton.setOnClickListener(v -> { /* ... */ });
contentView.addView(floatingButton, FragmentFloatingButton.createDefaultLayoutParams());

// 2. Show it — must be called explicitly, e.g. in onResume and onInsets
floatingButton.setButtonVisible(true, false);  // false = no animation on first show

// 3. Lift it above the navigation bar / tab bar
floatingButton.setTranslationY(-(navigationBarHeight + additionFloatingButtonOffset));
```

#### `setTranslationY` is overridden

`FragmentFloatingButton.setTranslationY(y)` does **not** call `super.setTranslationY(y)` directly. It stores `y` as `internalTranslationY` and calls `super.setTranslationY(y + additionalTranslationY)`, where `additionalTranslationY` is the show/hide slide offset managed by the animator. This means:
- Your translation and the show/hide animation compose correctly without interfering.
- Never call `super.setTranslationY` on a `FragmentFloatingButton` directly.

#### Visibility lifecycle

```java
// In onResume — ensure FAB is visible when the screen comes back into view
@Override
public void onResume() {
    super.onResume();
    if (floatingButton != null) floatingButton.setButtonVisible(true, false);
}

// In onInsets — reposition after insets are known
@Override
public void onInsets(int left, int top, int right, int bottom) {
    navigationBarHeight = bottom;
    checkUi_listViewPadding();
    if (floatingButton != null) {
        floatingButton.setTranslationY(-(navigationBarHeight + additionFloatingButtonOffset));
        floatingButton.setButtonVisible(true, false);
    }
}

// To hide (e.g. when scrolling down):
floatingButton.setButtonVisible(false, true);  // true = animate

// To show again:
floatingButton.setButtonVisible(true, true);
```

#### Layout params — use the standard factory methods

```java
// Standard 48×48 FAB (used by ContactsActivity, DialogsActivity):
FragmentFloatingButton.createDefaultLayoutParams()
// → 48×48, Gravity.BOTTOM|RIGHT, margin 20dp right, 14dp bottom

// Larger 56×56 variant:
FragmentFloatingButton.createDefaultLayoutParamsBig()
// → 56×56, same gravity and margins

// Sub-button (smaller, stacked above main FAB):
FragmentFloatingButton.createSubButtonLayoutParams()
// → 48×48, same gravity, 14dp bottom (sits above main FAB via setTranslationY)
```

Always use `createDefaultLayoutParams()` (not Big) unless the design specifically calls for a larger button — the standard size matches the Chats screen FAB exactly.

## 3. Logic / Architecture Rules

### 3.1 Multi-Account Architecture

**Every controller, storage, and config class is per-account. Always pass `currentAccount`.**

```java
// Correct — account-scoped:
MessagesController.getInstance(currentAccount).someMethod();
MessagesStorage.getInstance(currentAccount).someMethod();
ConnectionsManager.getInstance(currentAccount).sendRequest(...);
UserConfig.getInstance(currentAccount).getCurrentUser();

// In a BaseFragment subclass, use the inherited helpers:
getMessagesController().someMethod();
getMessagesStorage().someMethod();
getConnectionsManager().sendRequest(...);
getUserConfig().getCurrentUser();
```

- `currentAccount` is a field on `BaseFragment` and `BaseController`, defaulting to `UserConfig.selectedAccount`.
- `AccountInstance.getInstance(account)` is the service locator — it provides all controllers for a given account.
- `UserConfig.MAX_ACCOUNT_COUNT` = 4. Never assume a single account.

### 3.2 Network / API Rules

**Must follow: all Telegram API calls go through `ConnectionsManager.sendRequest`.**

```java
TLRPC.TL_someMethod req = new TLRPC.TL_someMethod();
req.someField = value;

getConnectionsManager().sendRequest(req, (response, error) -> {
    if (error == null) {
        TLRPC.TL_someResponse res = (TLRPC.TL_someResponse) response;
        AndroidUtilities.runOnUIThread(() -> {
            // update UI
        });
    } else {
        AndroidUtilities.runOnUIThread(() -> {
            BulletinFactory.of(MyFragment.this).showForError(error);
        });
    }
}, ConnectionsManager.RequestFlagFailOnServerErrors);
```

**Key rules:**
- The `RequestDelegate` callback runs on a background thread — always dispatch UI updates to the main thread via `AndroidUtilities.runOnUIThread`.
- Tag requests with `classGuid` so they are auto-cancelled when the fragment is destroyed:
  ```java
  getConnectionsManager().sendRequest(req, callback, 0, ConnectionsManager.DEFAULT_DATACENTER_ID, classGuid);
  ```
- Use `RequestFlagFailOnServerErrors` for most requests to get error callbacks on server errors.
- Use `RequestFlagWithoutLogin` only for pre-auth requests (login flow).
- TL schema classes live in `org.telegram.tgnet.TLRPC` and `org.telegram.tgnet.tl.*`.
- For typed requests use `sendRequestTyped(TLMethod<T>, Callback2<T, TL_error>)`.

**Higher-level operations — prefer controller methods over raw API calls:**
```java
// Prefer:
getMessagesController().loadFullUser(userId, classGuid, false);
// Over manually constructing TLRPC.TL_users_getFullUser
```

### 3.3 Local Storage Rules

**Two storage systems exist — use the right one:**

**1. SharedPreferences — for simple key/value settings:**

```java
// Global app settings (not per-account):
SharedConfig.someField = value;
SharedConfig.saveConfig();

// Per-account settings:
UserConfig config = getUserConfig();
config.someField = value;
config.saveConfig(false);

// Notification settings (per-account):
SharedPreferences prefs = MessagesController.getNotificationsSettings(currentAccount);
prefs.edit().putBoolean("key", value).apply();

// Custom preferences:
SharedPreferences prefs = ApplicationLoader.applicationContext
    .getSharedPreferences("myfeature", Context.MODE_PRIVATE);
```

**2. SQLite — for structured/relational data:**
- Use `MessagesStorage` for all message, dialog, user, and chat data.
- The custom wrapper is in `org.telegram.SQLite` — use `SQLiteDatabase`, `SQLiteCursor`, `SQLitePreparedStatement`.
- All database operations must run on `MessagesStorage`'s `storageQueue` (a `DispatchQueue`), never on the main thread.
- Never use Room, GreenDAO, or any other ORM.

```java
getMessagesStorage().storageQueue.postRunnable(() -> {
    SQLiteDatabase db = getMessagesStorage().getDatabase();
    // perform DB operations
    AndroidUtilities.runOnUIThread(() -> {
        // update UI with results
    });
});
```

### 3.4 Threading Rules

**Must follow: use `DispatchQueue` and `AndroidUtilities.runOnUIThread` — never use `AsyncTask`, `Thread`, or `Handler` directly for new code.**

```java
// Post to a named background queue:
Utilities.globalQueue.postRunnable(() -> {
    // background work
    AndroidUtilities.runOnUIThread(() -> {
        // UI update
    });
});

// Delayed UI update:
AndroidUtilities.runOnUIThread(() -> { ... }, 300);

// Cancel a delayed runnable:
AndroidUtilities.cancelRunOnUIThread(myRunnable);

// Create a dedicated queue for a feature:
DispatchQueue myQueue = new DispatchQueue("MyFeatureQueue");
myQueue.postRunnable(() -> { ... });
```

**Existing shared queues:**

| Queue | Use |
|---|---|
| `Utilities.globalQueue` | General background tasks |
| `Utilities.stageQueue` | Staging/processing tasks |
| `MessagesStorage.storageQueue` | All SQLite operations |
| `ConnectionsManager` (internal) | Network I/O (do not post to directly) |

**Thread safety rules:**
- UI reads/writes must be on the main thread.
- `@UiThread` annotations in `MessagesStorage` mark methods that must be called from the UI thread.
- Never block the main thread with I/O or heavy computation.

### 3.5 State Management / Event Bus Rules

**Must follow: use `NotificationCenter` for cross-component state updates.**

```java
// Subscribe (in onFragmentCreate):
getNotificationCenter().addObserver(this, NotificationCenter.chatInfoDidLoad);

// Unsubscribe (in onFragmentDestroy — must mirror every addObserver):
getNotificationCenter().removeObserver(this, NotificationCenter.chatInfoDidLoad);

// Implement the interface:
@Override
public void didReceivedNotification(int id, int account, Object... args) {
    if (id == NotificationCenter.chatInfoDidLoad) {
        TLRPC.ChatFull info = (TLRPC.ChatFull) args[0];
        // update UI
    }
}

// Post a notification (from a controller):
getNotificationCenter().postNotificationName(NotificationCenter.chatInfoDidLoad, chatFull);
```

- All `NotificationCenter` constants are defined as `public static final int` fields in `NotificationCenter.java`.
- When adding a new event, add a new constant to `NotificationCenter` following the `totalEvents++` pattern.
- Notifications are delivered on the main thread.
- Never invent a custom observer/listener pattern if `NotificationCenter` covers the use case.

### 3.6 Controller / Service Access Rules

**Must follow: access all services through `AccountInstance` or the `BaseFragment`/`BaseController` helper methods.**

```java
// From a BaseFragment:
getMessagesController()     // MessagesController
getMessagesStorage()        // MessagesStorage
getConnectionsManager()     // ConnectionsManager
getNotificationCenter()     // NotificationCenter
getUserConfig()             // UserConfig
getContactsController()     // ContactsController
getMediaDataController()    // MediaDataController
getSendMessagesHelper()     // SendMessagesHelper
getFileLoader()             // FileLoader
getDownloadController()     // DownloadController
```

- Never call `MessagesController.getInstance(0)` with a hardcoded account number.
- Never instantiate controllers directly — they are all singletons accessed via `getInstance(account)`.
- New controllers must extend `BaseController` and follow the double-checked locking singleton pattern used by all existing controllers.

### 3.7 Error Handling Rules

- Network errors: check `error != null` in `RequestDelegate`, show via `BulletinFactory.of(fragment).showForError(error)`.
- Exceptions in background threads: catch and log with `FileLog.e(e)`.
- Exceptions in UI code: catch and log with `FileLog.e(e)` — never swallow silently.
- Never show `android.widget.Toast` for errors — use `BulletinFactory`.
- For critical errors that require user action, use `AlertDialog.Builder`.

```java
// Logging:
FileLog.d("tag: debug message");
FileLog.e(exception);
FileLog.e("tag: error message");
```

### 3.8 Serialization Rules

- All Telegram API objects serialize/deserialize via `TLObject.serialize(SerializedData)` / `TLObject.readParams(SerializedData, boolean)`.
- For persisting TL objects to SharedPreferences, use `SerializedData` + `Base64`:
  ```java
  SerializedData data = new SerializedData(obj.getObjectSize());
  obj.serializeToStream(data);
  String encoded = Base64.encodeToString(data.toByteArray(), Base64.DEFAULT);
  prefs.edit().putString("key", encoded).apply();
  ```
- For JSON, use `org.json.JSONObject` / `JSONArray` (already a dependency). Gson is available but used sparingly (mainly in `FileLog`).
- Never use Java serialization (`Serializable`).

## 5. Implementation Constraints for AI

### Hard Constraints (never violate)

1. **No XML layouts for screens.** All UI is built programmatically in `createView()`.
2. **No Jetpack Compose.** The entire UI stack is custom Android Views.
3. **No `android.app.AlertDialog`.** Use `org.telegram.ui.ActionBar.AlertDialog`.
4. **No `android.widget.Toast`.** Use `BulletinFactory`.
5. **No `AsyncTask`, `Thread`, or `Handler` for new background work.** Use `DispatchQueue` or `AndroidUtilities.runOnUIThread`.
6. **No hardcoded colors.** All colors via `Theme.getColor(Theme.key_*)`.
7. **No hardcoded dp/px values.** All dimensions via `AndroidUtilities.dp()`.
8. **No hardcoded strings.** All user-visible text via `LocaleController.getString(R.string.*)`.
9. **No direct `RecyclerView`.** Use `RecyclerListView`.
10. **No Room, GreenDAO, or ORM.** Use `MessagesStorage` and the custom `org.telegram.SQLite` wrapper.
11. **No Retrofit, OkHttp, or any HTTP client for Telegram API.** Use `ConnectionsManager.sendRequest`.
12. **No hardcoded account index (e.g., `getInstance(0)`).** Always use `currentAccount`.
13. **No `dialog.show()` from a fragment.** Use `showDialog(dialog)` from `BaseFragment`.
14. **No new singleton pattern.** Follow the double-checked locking pattern used by all existing controllers.
15. **No Kotlin coroutines.** The codebase is Java; threading is done via `DispatchQueue`.

### Soft Constraints / Preferred Practices

- Prefer `UniversalFragment` + `UniversalAdapter` + `UItem` for new settings-style screens over the classic row-index adapter pattern.
- Prefer `BottomSheetWithRecyclerListView` over `BottomSheet` for sheets with scrollable content.
- Prefer `ItemOptions` over `ActionBarPopupWindow` for item-level context menus.
- Pass `resourcesProvider` through all cell constructors to support per-fragment theme overrides.
- Use `classGuid` to tag all network requests so they auto-cancel on fragment destroy.
- Keep row index fields as `private int` fields on the fragment, recomputed in a `updateRows()` method.
- Use `@Keep` annotation on adapter `onBindViewHolder` methods and fields accessed by reflection if ProGuard is active.

### Forbidden Anti-Patterns

- Creating a new `Activity` for a feature screen (use `BaseFragment` + `presentFragment`).
- Storing UI state in static fields (use fragment fields or `NotificationCenter`).
- Calling `notifyDataSetChanged()` on every minor update — prefer `notifyItemChanged`, `notifyItemInserted`, etc. where possible.
- Performing network or database I/O on the main thread.
- Using `System.out.println` or `android.util.Log` — use `FileLog`.
- Importing or adding new third-party libraries without necessity — the dependency set is intentionally minimal.
- Creating parallel implementations of existing infrastructure (e.g., a second image loader, a second DB layer).
- Using `Fragment` transactions or the Android back stack — the app has its own stack in `ActionBarLayout`.

## 6. Feature Implementation Workflow

Follow this process before writing any new code:

### Step 1 — Find the closest existing screen
Search `org.telegram.ui` for a screen that does something similar. Read its `createView`, adapter, and `didReceivedNotification` implementations. Your new screen should mirror its structure.

### Step 2 — Identify required cells
List the row types your screen needs. Map each to an existing cell in `org.telegram.ui.Cells`. Only create a new cell class if no existing cell can be adapted with minor changes.

### Step 3 — Identify required notifications
Search `NotificationCenter` for existing events that cover your data changes. Only add a new `NotificationCenter` constant if no existing event fits.

### Step 4 — Identify required controller methods
Search `MessagesController`, `MediaDataController`, `ContactsController`, etc. for existing methods that perform the needed operation. Only add new controller methods if the operation is genuinely new.

### Step 5 — Identify required API calls
Find the TL method in `TLRPC` or `org.telegram.tgnet.tl.*`. Construct the request object, send via `ConnectionsManager.sendRequest`, handle the response on the UI thread.

### Step 6 — Wire up the fragment
1. Extend `BaseFragment` (or `UniversalFragment` for settings-style screens).
2. Implement `onFragmentCreate` — subscribe to `NotificationCenter`, load initial data.
3. Implement `createView` — set up `actionBar`, create `RecyclerListView`, set adapter.
4. Implement `updateRows()` — compute row indices.
5. Implement `ListAdapter` inner class (or use `UniversalAdapter`).
6. Implement `didReceivedNotification` — react to data changes, call `updateRows()` + `adapter.notifyDataSetChanged()`.
7. Implement `onFragmentDestroy` — unsubscribe from `NotificationCenter`.

### Step 7 — Navigate to the new screen
Add `presentFragment(new MyFeatureActivity())` at the appropriate call site (e.g., in an existing screen's `onItemClick`).

### Step 8 — Validate consistency
- No hardcoded colors, dimensions, or strings.
- All background work off the main thread.
- All UI updates on the main thread.
- `resourcesProvider` passed to all cells.
- Network requests tagged with `classGuid`.
- `NotificationCenter` observers balanced (add = remove).
