## 1. Project Meta & Golden Rules for AI
**Project Name:** Bonyan (Comprehensive Family & Mission Tracking System)
**Base Codebase:** Telegram Android Client
**Core Philosophy:** Offline-first, Telegram-native UX, highly modular.

### STRICT AI DIRECTIVES (DO NOT VIOLATE)
1. **NO CORE FORKING:** Do not alter the internal logic of `org.telegram.ui`, `org.telegram.messenger`, or `org.telegram.tgnet`.
   - *Exception:* You may add minimal layout injections in `LaunchActivity.java` (for the Bottom Navigation) and `ActionBar` (for top-right buttons). Keep these changes under 50 lines total.
2. **MINIMIZE THE CHANGES TO TELEGRAM CODE:** NEVER rename `org.telegram.*` constants, variables, functions, etc., even when their meanings are changed in Bonyan. The changed meanings can be reflected in the UI using the localized strings.
3. **ISOLATION:** All new code MUST reside in the `org.bonyan.*` package within the `TMessagesProj` module. Use inheritance, composition, and Telegram's existing base classes (`BaseFragment` (via `BonyanBaseFragment`), `RecyclerView`, `ActionBar`).
4. **OFFLINE-FIRST:** The UI must never block on network calls. All UI reads from the local SQLite database. Network calls only update the local DB and trigger UI refreshes via `NotificationCenter`.
5. **REUSE OVER REWRITE:** Before creating a custom UI component or layout, check if Telegram has an existing one (e.g., `TextSettingsCell`, `HeaderCell`, `DialogCell`, `ChatMessageCell`). Reuse them. NEVER use standard Android components, unless absolutely necessary AND after asking for my confirmation. Particularly, use `BulletinFactory` for in-app toast-style notifications instead of the `android.widget.Toast`. ALWAYS make sure the required java imports are added and correct.
6. **UNDO MECHANISM:** All destructive actions must be routed through `BonyanUndoManager` to trigger the global 10-second Snackbar.

### Other Notes
- **Develop firstly for English language.** All strings and code comments must be in English, even if the AI prompt is in another language. If you come across any Persian comment, translate it to English. Translation to other languages will be performed later.
- When you add a string to an xml file, always check the key is unique.

### The Bridge Pattern (Dependency Inversion)

To avoid circular dependencies while keeping Telegram core modifications minimal:

1. **Interface in Telegram Core:** `org.telegram.ui.IBonyanEntryPoint` вЂ” a minimal interface that `MainTabsActivity` can call.
2. **Implementation in Bonyan:** `org.bonyan.ui.BonyanEntryPointImpl` вЂ” implements the interface, instantiates Bonyan fragments.
3. **Registry (Static Holder):** `org.bonyan.data.BonyanRegistry` вЂ” holds the singleton instance of the entry point.
4. **Auto-Initialization:** `org.bonyan.data.BonyanInitProvider` вЂ” a `ContentProvider` that runs before `Application.onCreate()` and registers the entry point.

```java
// In MainTabsActivity.java (Telegram core, minimal change)
@Override
protected BaseFragment createBaseFragmentAt(int position) {
    if (position >= 0 && position <= 3) {
        IBonyanEntryPoint bonyan = BonyanRegistry.get();
        if (bonyan != null) {
            BaseFragment fragment = bonyan.getFragment(position);
            if (fragment != null) return fragment;
        }
    }
    // ... original Telegram logic as fallback
}
```

**Why This Works:**
- `MainTabsActivity` only imports `IBonyanEntryPoint` and `BonyanRegistry`
- No direct imports of `org.bonyan.ui.*` classes in Telegram core
- All Bonyan fragment instantiation logic lives in `BonyanEntryPointImpl`
- Easy to merge future Telegram updates (only ~10 lines to resolve)

## 4. Phase 0: Core Infrastructure Setup
**Goal:** Establish the offline-first foundation and the bridge pattern without breaking Telegram's core.

### 4.1 Bridge Pattern Implementation
1. Create `org.telegram.ui.IBonyanEntryPoint` interface in `TMessagesProj`.
2. Create `org.bonyan.data.BonyanRegistry` static holder in `org.bonyan.*`.
3. Create `org.bonyan.data.BonyanInitProvider` (ContentProvider) in `org.bonyan.*`.
4. Register the ContentProvider in `TMessagesProj/src/main/AndroidManifest.xml`.
5. Create `org.bonyan.ui.BonyanEntryPointImpl` implementing the interface.

### 4.2 MainTabsActivity Modification (Minimal)
1. Modify `MainTabsActivity.createBaseFragmentAt()` to delegate to `BonyanRegistry` for positions 0-3.
2. Update tab titles: "Missions", "Planner", "Family", "Profile".
3. Update tab icons to appropriate Material icons.
4. **Keep original Telegram logic as fallback** for positions outside 0-3.

### 4.3 Database & Sync Engine
1. Initialize Room Database in `BonyanInitProvider`.
2. Create `BonyanSyncWorker` (using `WorkManager`) to process the `SyncQueue` table in batches when online.

### 4.4 Undo Snackbar System
1. Create `BonyanUndoManager`. It holds a reference to the last destructive action (Runnable).
2. When triggered, it shows a Snackbar with a 10-second progress bar. If clicked, it executes the undo Runnable.

## 5. Phase 1: Personal & Family Modules
**Goal:** Identity, family tree, and offline-first contact management.

### Step 1.1: Profile & Settings (Profile)
- **UI:** Create `BonyanProfileFragment`. Use `TextSettingsCell` for fields.
- **Logic:**
  - Minimum fields: Name, Gender, Birth Year.
  - Implement a dynamic Tag system (`PersonTag` table) for extensible attributes (Skills, Economic, Media, etc.).
  - All edits save to Local DB immediately and push to `SyncQueue`.

### Step 1.2: Family & Contacts (Family)
- **UI:** Create `BonyanFamilyFragment` extending the logic of `ContactsActivity`.
- **Logic:**
  - Display persons the user has access to.
  - **Advanced Filter:** The top-right settings button opens a bottom sheet to filter by Tags, Location, Skills.
  - **Claim Relationship:** Add a "Claim Relationship" button on a person's profile.
    - *Offline logic:* Saves to `FamilyRelation` with `sync_status = PENDING`.
    - *Sync logic:* Sends request to server. When the other party confirms, updates local DB.

### Step 1.3: Registration & The "Public Mission"
- **Flow:** Phone login -> Enter Referrer Phone -> Wait for approval.
- **Implementation:**
  - Create a system-level "Public Mission". Every registered user is auto-joined.
  - Registration approval requests are injected as "Notifications" into this Public Mission's chat feed.

## 7. Execution Workflow for AI IDE

When generating code for this project, follow this exact sequence:

1. **Context Gathering:** Before writing a new Fragment, read the corresponding Telegram base class (e.g., read `DialogsActivity.java` before writing `BonyanMissionListFragment`).
2. **Package Check:** Confirm the code belongs in the `org.bonyan.*` package within `TMessagesProj`. All UI classes go directly into `org.bonyan.ui` (no sub-packages).
3. **Layout First:** Define the XML layout using Telegram's existing views (`org.telegram.ui.Cells.*`, `org.telegram.ui.Components.*`).
4. **Data Layer:** Create the Room Entity and DAO if it doesn't exist.
5. **Repository:** Write the offline-first repository method (Read from DB -> Return to UI -> Trigger Sync in background).
6. **UI Binding:** Bind the local data to the UI using Telegram's `RecyclerListView` and `BaseFragment` lifecycle methods (`onCreateView`, `onResume`).
7. **NotificationCenter:** Use `NotificationCenter.getInstance().postNotificationName(...)` to update the UI when the sync engine updates the local DB.

## 8. Build Performance Rules

To ensure fast incremental builds:

1. **Never modify `ApplicationLoader.java`** or any other Telegram class referenced by hundreds of files.
2. **Use ContentProvider** for Bonyan initialization (auto-runs before `Application.onCreate()`).
3. **Keep Bonyan code isolated** in `org.bonyan.*` package to minimize cross-references.
4. **Test incremental builds** after each major change:
   - Modify a Bonyan file в†’ ideally only Bonyan files should recompile.
   - Avoid modifying Telegram classes that are imported by many other files.
