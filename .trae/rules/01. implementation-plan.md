# Bonyan (بنیان) - Implementation Plan

## 1. Project Meta & Golden Rules for AI
**Project Name:** Bonyan (Comprehensive Family & Mission Tracking System)
**Base Codebase:** Telegram Android Client
**Core Philosophy:** Offline-first, Telegram-native UX, highly modular.

### STRICT AI DIRECTIVES (DO NOT VIOLATE)
1. **NO CORE FORKING:** Do not alter the internal logic of `org.telegram.ui`, `org.telegram.messenger`, or `org.telegram.tgnet`.
   - *Exception:* You may add minimal layout injections in `LaunchActivity.java` (for the Bottom Navigation) and `ActionBar` (for top-right buttons). Keep these changes under 50 lines total.
2. **MINIMIZE THE CHANGES TO TELEGRAM CODE:** **NEVER** rename `org.telegram.*` constants, variables, functions, etc, even when their meanings are changed in Bonyan. The changed meanings can be reflected in the UI using the localized strings.
2. **ISOLATION:** All new code MUST reside in the `org.bonyan.*` package. Use inheritance, composition, and Telegram's existing base classes (`BaseFragment`, `RecyclerView`, `ActionBar`).
3. **OFFLINE-FIRST:** The UI must **never** block on network calls. All UI reads from the local SQLite database. Network calls only update the local DB and trigger UI refreshes via `NotificationCenter`.
4. **REUSE OVER REWRITE:** Before creating a custom UI component, check if Telegram has an existing one (e.g., `TextSettingsCell`, `HeaderCell`, `DialogCell`, `ChatMessageCell`). Reuse them. *NEVER* use standard Android components, unless absolutely necessary *AND* after asking for my confirmation. Particularly, use `BulletinFactory` for in-app toast-style notifications instead of the `android.widget.Toast`.
5. **UNDO MECHANISM:** All destructive actions must be routed through `BonyanUndoManager` to trigger the global 10-second Snackbar.

### Other Notes
- Develop firstly for English language. All strings and code comments must be in English, even if the AI prompt is in another language. If you come across any Persian comment, translate it to English. Translation to other languages will be performed later.
- When you add a string to an xml file, always check the key is unique.

## 2. Architectural Blueprint

### Package Structure
```text
org.bonyan/
├── data/
│   ├── local/         # Room/SQLite Database, DAOs, Entities
│   ├── remote/        # Retrofit/OkHttp API interfaces for Bonyan Backend
│   ├── repository/    # Offline-first Repositories (Local DB + Sync)
│   └── sync/          # Background Sync Engine (WorkManager)
├── domain/
│   ├── model/         # Pure Kotlin/Java domain models
│   └── usecase/       # Business logic (e.g., ReputationCalculator)
├── ui/
│   ├── base/          # Base fragments, custom views, UndoManager
│   ├── family/        # Family tree, contacts, profile
│   ├── mission/       # Mission list, details, creation
│   ├── calendar/      # Calendar and scheduling
│   └── org/           # Organization management
└── util/              # Bonyan-specific utilities
```

### Local Database Schema (SQLite / Room)
*AI: Generate Room Entities and DAOs for the following. Use `last_synced_at` for delta syncing.*
- **`Person`**: id, name, gender, birth_year, phone, national_id, avatar_url, reputation_score.
- **`PersonTag`**: person_id, tag_category (e.g., 'SKILL', 'ECONOMIC'), tag_value.
- **`FamilyRelation`**: id, person_a_id, person_b_id, relation_type, sync_status (PENDING/SYNCED).
- **`Organization`**: id, name, type, default_ranks.
- **`OrgMembership`**: org_id, person_id, rank, local_reputation, sync_status.
- **`Mission`**: id, title, description, logo_url, status (DRAFT/ACTIVE/ARCHIVED), parent_mission_id, org_id, start_time, end_time, priority_color.
- **`MissionComponent`**: id, mission_id, type (TASK/ITEM/COUNTER/NOTE/POLL/FEEDBACK), title, data_json.
- **`MissionParticipant`**: mission_id, person_id, role (ADMIN/ANALYST/FOLLOWER).
- **`CalendarEvent`**: id, mission_id, event_time, duration, is_overdue, reminder_type (EXACT/AZAN/LEITNER/VAGUE).
- **`SyncQueue`**: id, entity_type, entity_id, action (INSERT/UPDATE/DELETE), payload_json, retry_count.

## 4. Phase 0: Core Infrastructure Setup
*Goal: Establish the offline-first foundation and global UI hooks without breaking Telegram's core.*

1. **Bottom Navigation Injection:**
   - Modify `LaunchActivity.java` minimally: Add a `<org.bonyan.ui.base.BonyanBottomNav>` to the root layout.
   - Intercept fragment transactions to swap between `BonyanMissionListFragment`, `BonyanCalendarFragment`, `BonyanFamilyFragment`, and `BonyanProfileFragment`.
2. **Global Top-Right Buttons:**
   - Extend Telegram's `ActionBar`. Add a global `Settings` (filter) button and `Search` button to the right side.
3. **Undo Snackbar System:**
   - Create `BonyanUndoManager`. It holds a reference to the last destructive action (Runnable).
   - When triggered, it shows a Snackbar with a 10-second progress bar. If clicked, it executes the undo Runnable.
4. **Database & Sync Engine:**
   - Initialize Room Database.
   - Create `BonyanSyncWorker` (using `WorkManager`) to process the `SyncQueue` table in batches when online.

## 5. Phase 1: Personal & Family Modules
*Goal: Identity, family tree, and offline-first contact management.*

### Step 1.1: Profile & Settings (نمایه و تنظیمات)
- **UI:** Create `BonyanProfileFragment`. Use `TextSettingsCell` for fields.
- **Logic:**
  - Minimum fields: Name, Gender, Birth Year.
  - Implement a dynamic Tag system (`PersonTag` table) for extensible attributes (Skills, Economic, Media, etc.).
  - All edits save to Local DB immediately and push to `SyncQueue`.

### Step 1.2: Family & Contacts (صفحه خانواده)
- **UI:** Create `BonyanFamilyFragment` extending the logic of `ContactsActivity`.
- **Logic:**
  - Display persons the user has access to.
  - **Advanced Filter:** The top-right settings button opens a bottom sheet to filter by Tags, Location, Skills.
  - **Claim Relationship:** Add a "Claim Relationship" button on a person's profile.
    - *Offline logic:* Saves to `FamilyRelation` with `sync_status = PENDING`.
    - *Sync logic:* Sends request to server. When the other party confirms, updates local DB.

### Step 1.3: Registration & The "Public Mission"
- **Flow:** Phone login -> Enter Referrer Phone -> Wait for approval.
- **Implementation:**
  - Create a system-level "Public Mission" (ماموریت عمومی). Every registered user is auto-joined.
  - Registration approval requests are injected as "Notifications" into this Public Mission's chat feed.

## 6. Phase 2: Organization & Mission Tracking
*Goal: The operational engine. Missions, reputation, and calendar.*

### Step 2.1: Organization & Reputation (تشکیلات و منزلت)
- **Logic:**
  - Implement `ReputationCalculator`.
  - Base score = minimum for rank (e.g., Newcomer=1, In the Field=500).
  - *Cross-Org Bonus:* If user has ≥200 rep in another org, add +100 to base.
  - *Feedback Loop:* +10 for standard likes. Admin likes = variable. Dislike = -2 to target, -1 to giver. Prevent spam via frequency checks.
- **UI:** Show reputation badge on `BonyanProfileFragment` and mission participant lists.

### Step 2.2: Mission Core & Components (ماموریت‌ها)
- **Data:** Highly relational. `Mission` -> `MissionComponent` (Tasks, Items, Counters, Notes, Polls, Feedback).
- **Creation:** Build `BonyanMissionCreateActivity`. Reuse Telegram's channel creation wizard UI. Allow toggling specific components.
- **Status Workflow:** Draft (Validation) -> Active -> Archived.

### Step 2.3: Mission List & Explore (صفحه ماموریت‌ها و ویترین)
- **UI:** `BonyanMissionListFragment`. Reuse `DialogsActivity` layout.
- **FABs:**
  - **Blue FAB (Bottom Right):** Create new mission.
  - **Yellow FAB (Above Blue):** Open "Explore" (ویترین). Fetches active missions from server based on user's org, rank, and tags.
- **List Item:** Custom `BonyanMissionCell` (extends `DialogCell`). Shows logo, title, last notification summary, time, unread badge, priority color, and progress %.

### Step 2.4: Mission Details & Interaction (The "Chat" Paradigm)
- **UI:** `BonyanMissionDetailFragment`. Reuse `ChatActivity` layout.
- **Logic:**
  - "Messages" are Notifications. Use `ChatMessageCell`. Clearly label if the sender is a User or Subsystem.
  - **Header Dashboard:** Clicking the top bar opens a dashboard (reuse Telegram's profile/media tabs) showing stats and a "Participate" button.
  - **Participate Drawer:** Bottom sheet listing components. User can log progress, set personal priorities, and add personal tags.
  - Clicking a notification related to a component opens the specific UI (reuse Telegram's Poll/Note views).

### Step 2.5: Calendar & Scheduling (صفحه تقویم)
- **UI:** `BonyanCalendarFragment`.
- **Top Section:** Month/Year selector, "Today" button, horizontal scrollable week view (expandable to month).
- **Main Section:** `RecyclerView` with sticky headers for Days.
- **Logic:**
  - Show mission events chronologically. Overdue events at the top.
  - **Drag & Drop:** Implement `ItemTouchHelper` to drag events between days.
  - **Time Pickers:** Custom pickers for Azan times, Leitner intervals, and vague times (Morning, Afternoon, etc.).

## 7. Execution Workflow for AI IDE

When generating code for this project, follow this exact sequence:

1. **Context Gathering:** Before writing a new Fragment, read the corresponding Telegram base class (e.g., read `DialogsActivity.java` before writing `BonyanMissionListFragment`).
2. **Layout First:** Define the XML layout using Telegram's existing views (`org.telegram.ui.Cells.*`, `org.telegram.ui.Components.*`).
3. **Data Layer:** Create the Room Entity and DAO if it doesn't exist.
4. **Repository:** Write the offline-first repository method (Read from DB -> Return to UI -> Trigger Sync in background).
5. **UI Binding:** Bind the local data to the UI using Telegram's `RecyclerListView` and `BaseFragment` lifecycle methods (`onCreateView`, `onResume`).
6. **NotificationCenter:** Use `NotificationCenter.getInstance().postNotificationName(...)` to update the UI when the sync engine updates the local DB.
